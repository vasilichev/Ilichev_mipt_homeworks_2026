<img src="./_resources/breaker.jpg" height=260>

В рамках ДЗ вам предстоит познакомиться с паттерном [CircuitBreaker](https://martinfowler.com/bliki/CircuitBreaker.html). Часто в программах встречаются запросы к другим системам, и иногда эти системы могут быть недоступны (авария, плановые работы, ограничененные ресурсы и т.д.). Бомбардировка упавшей системы новыми запросами не способствует её восстановлению, к тому же каждый новый запрос занимает время (особенно если падаем по Timeout), что отнимает ресурс и у нашей системы. Именно поэтому обращения к таким системам можно и нужно оборачивать в некоторый "Автомат", который в обычном режиме пропускает все запросы, а в случае появления ошибок <u>на время</u> прекращает выполнение запросов.

## Задача

Требуется разработать класс `CircuitBreaker`, экземпляр которого можно будет использовать, как декоратор для функций, которые выполняют внешние запросы.

```python
import requests

cbreaker = CircuitBreaker()

@cbreaker
def external_request(comment_id: int):
    requests.get(f"https://helloworld.org/comments/{comment_id}")
```

```python
import requests

cbreaker = CircuitBreaker(critical_count=10, triggers_on=ConnectionError)

@cbreaker
def external_request():
    requests.get("https://helloworld.org/")
```

```python
import requests

cbreaker = CircuitBreaker(critical_count=10, time_to_recover=30)

@cbreaker
def external_request():
    requests.get("https://helloworld.org/")
```



Таким образом, декоратор должен быть параметризируемым и поддерживать настройку:
1. `critical_count` - Если запрос critical_count раз подряд упал с ошибкой, автомат срабатывает и выбрасывает своё исключение `BreakerError`. Если запрос прошёл успешно, счётчик неудачных подключений сбрасывается. Аргумент `critical_count` по умолчанию должен иметь значение 5. Он должен быть целочисленным и строго положительным, иначе создаём исключение ValueError с сообщением "Breaker count must be positive integer!"
2. `time_to_recover` - Время деактивации функции. Спустя time_to_recover секунд нужно перестать блокировать функцию и снова начать пропускать запросы. Значение по умолчанию - 30. Аргумент должен быть целочисленным и строго положительным, иначе создаём ValueError с сообщением "Breaker recovery time must be positive integer!"
3. `triggers_on` - Тип исключения, на которое должен реагировать автомат. Например, можно явно указать TimeoutError. Если аргумент не задан, реагируем на любое исключение-потомка Exception.

ВАЖНО: Не нужно бросать ValueError по одиночке - в этой задаче для проброса ошибок валидации нужно использовать ExceptionGroup с вложенным/и ValueError и отдельным сообщением "Invalid decorator args."


## Несколько замечаний о вашем BreakerError
* Ваше кастомное исключение должно хранить в себе дополнительные атрибуты: `func_name` и `block_time` - название декорируемой функции (в формате <имя модуля>.<имя функции>) и дату-время блокировки автомата (например, 2026-05-15 10:00:00) в таймзоне UTC.
* Если BreakerError порождается другим исключением, обязательно добавьте его, как источник!
* В качестве сообщения BreakerError должен содержать строку: "Too much requests, just wait."


## На что обратить внимание
* Нужно уметь декорировать функции с разным числом и типами аргументов - не забывайте передавать их в декорируемую функцию.
* Допустим, critical_count = 3 и запросы летят в упавший сервис. Это значит, что два раза мы получим оригинальную ошибку (например, TimeoutError), а на третий раз ошибка уже идёт от декоратора (BreakerError с источником TimeoutError).
* Пока автомат не закрыт, пойманные исключения нужно пробрасывать выше - не важно, это целевой тип (из triggers_on) или нецелевой.
* Для примера внешнего вызова в файле hw67.py есть функция get_comments - тесты явно применяют код вашего декоратора к этой функции, поэтому ставить `@cbreaker` над этой функцией НЕ НУЖНО.
